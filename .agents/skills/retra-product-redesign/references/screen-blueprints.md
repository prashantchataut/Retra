# Screen Blueprints

## Onboarding

Three stages are sufficient:

1. Explain Retra as a memory player.
2. Explain user-owned imports, folder scanning, and patch workflows.
3. Explain offline-first behavior and optional identity.

A user must be able to complete onboarding without signing in. Do not request storage permission before explaining the value and before the user chooses an import action.

## Home

Order content by return intent:

1. Continue-playing hero.
2. Recent and favorite worlds.
3. Small personal library snapshot.
4. Trust/privacy reassurance.

Do not repeat the entire Library grid on Home.

## Library

- Artwork-first adaptive grid by default; detailed list remains available.
- Search title, aliases, tags, and collections.
- Filters: all, continue, favorites, patched, homebrew, unplayed.
- Keep import file and scan folder visible without dominating the screen.
- Empty state teaches supported file and patch workflows.

## Discover

- Lead with creator source and verification language.
- Separate creator-page links from checksum-verified downloadable releases.
- Show download, verify, import progress as one coherent action.
- Make failed validation understandable; never silently downgrade trust.

## Profile

- Treat Profile as a first-class destination.
- Show local identity, play progress, achievements, points, and save-vault summary.
- Explain exactly what is shareable and what remains local.
- Avoid invented public rankings or social activity.

## Settings

Group by user intent:

- Look & feel
- Library
- Gameplay
- Controls
- Sound & alerts
- Privacy

Use progressive disclosure and human-readable labels. Avoid a single endless technical list.

## Player

- Game canvas is primary.
- Quick save/load, pause, fast-forward, and control-layout actions belong in a compact overlay.
- Persist per-game overrides separately from global defaults.
- Make destructive save actions explicit and reversible when possible.
